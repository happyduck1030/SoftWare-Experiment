力资源管理系统需求文档（文字描述版）

本系统用于管理企业的人力资源信息，主要包括组织架构、员工档案和薪酬管理三大模块。系统支持两类用户角色：管理员（admin）和员工（employee）。员工中又细分为普通员工和机构负责人（boss），后者在其所负责的机构范围内拥有有限管理权限。

所有用户登录后，必须能查看自己所属的完整机构路径（一级机构 → 二级机构 → 三级机构），并能看到其上级机构（即上一层级）的负责人（boss）是谁。这一信息应在登录接口中一并返回，供前端展示。

管理员拥有全部权限，可进行以下操作：

设置和维护三级机构树形结构，并为每个三级机构指定一名负责人；
管理职位，每个职位必须归属于一个三级机构；
定义薪酬项目（如基本工资、绩效等），设置其类型和状态；
登记、复核、查询、变更和删除任意员工的人事档案；
制定、审核、调整薪酬标准，并按月登记、复核和查询薪酬发放记录。
普通员工仅能查看和有限更新自己的信息：

可查看个人档案（不含他人数据）；
可查看本人历史薪酬发放明细；
可通过专用接口更新个人非敏感信息（如联系电话、邮箱等）；
不可访问系统设置、他人档案或薪酬数据。
机构负责人（即某三级机构的 boss）除具备普通员工的所有能力外，还可在其负责的单一三级机构范围内执行以下操作：

查看该机构下所有下属员工的基本档案（不含身份证号等敏感字段）；
更新下属员工的部分非核心字段（如联系电话、工作状态等，具体字段由业务白名单控制）；
查看本机构员工的薪酬发放汇总列表（默认显示姓名、月份、总金额，不强制展示明细）；
不可操作系统设置、薪酬标准、机构结构等全局功能，也不可跨机构操作。
权限控制必须在后端严格实施：

所有 API 接口需验证 JWT 登录状态；
涉及写操作或管理功能的接口，仅允许 admin 或符合条件的机构 boss 访问；
查询他人数据时，必须校验当前用户是否为 admin，或目标员工是否属于其管辖机构且当前用户是该机构 boss；
禁止通过 URL 参数、请求体或前端传参绕过权限检查；
敏感字段（如身份证、银行账号、详细薪酬）应根据角色和归属关系动态过滤。
关键业务规则：

员工与三级机构绑定，不可无归属；
薪酬标准基于“职位 + 三级机构”组合设定；
薪酬发放按“年月 + 三级机构”生成批次，由管理员操作并复核；
删除操作建议采用软删除（标记 is_deleted），保留审计痕迹；
所有重要操作（如审核、删除、发放）应记录操作人、时间及目标对象。
系统设计需保证可扩展性，当前仅实现 admin 和 employee（含 boss）两级权限模型，但架构应支持未来引入更多角色（如 HR 专员）。前后端开发均须以此文档为依据，确保功能一致性和安全性。
